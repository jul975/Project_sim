Do You Have a Determinism-Breaking Bug Right Now?
--------------------------------------------------

Not necessarily, if all of the following hold:
- You never use `master_ss` after initialization.
- You never call `agent_seed.spawn()` after initialization.
- Current determinism tests (forward clone + full trajectory) pass.

Do You Have a Future Risk?
--------------------------

Yes.

As soon as you add new streams via `.spawn()` on reconstructed `SeedSequence` objects (`agent_seed`, `master_ss`), the current reconstruction scheme can diverge.

Also:
- `schema v1` may fail to catch environment-level divergence.

How to Make This Future-Proof Without Breaking Lineage
------------------------------------------------------

Choose and enforce one invariant strategy.

Option 1: Explicit Lineage Only (Recommended)
---------------------------------------------

Invariant:
- After initialization, never call `.spawn()` on `SeedSequence` objects that must survive clone/restore.

For new streams (for example, combat/weather), derive seeds from stable keys:

```python
SeedSequence(entropy=agent_entropy, spawn_key=agent_spawn_key + (STREAM_TAG,))
```

or:

```python
spawn_key + (STREAM_TAG, stream_index)
```

Rules:
- Track only explicit counters (for example, `agent_spawn_count`).
- Include those counters in snapshot/hashing.
- Treat `agent_seed` as init-only splitter.
- Treat `master_ss` as init-only.

This matches the current mental model.

Option 2: True SeedSequence Continuation
----------------------------------------

If you want to call `.spawn()` later on `agent_seed` or `master_ss`, you must:
- Track "children spawned so far" counters yourself.
- Restore by recreating the `SeedSequence`, then advance it (`spawn(k)` and discard).
- Include these counters in snapshot state.

Important:
- Current "append `spawn_count` to `spawn_key`" is not equivalent to true continuation.
